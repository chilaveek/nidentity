=========================================================
СТРУКТУРА ПРОЕКТА NIDSHOP И ОПИСАНИЕ АЛГОРИТМОВ
=========================================================

Проект разделен на две основные части: клиентское приложение (клиент) и серверное приложение (бэкенд).

---------------------------------------------------------
1. КЛИЕНТСКАЯ ЧАСТЬ (Frontend)
---------------------------------------------------------
Путь: d:\_учеба\веб-приложения\nidshop\client
Стек: React, TypeScript, Vite.

Полная структура папок и ключевых файлов:
client/

└── src/

    ├── components/         - Переиспользуемые UI компоненты
    │   ├── admin/          - Компоненты для админки (AdminSidebar.tsx)
    │   ├── author/         - Компоненты для кабинета автора (AuthorSidebar.tsx)
    │   ├── figma/          - Экспортированные SVG компоненты из дизайна
    │   ├── guards/         - Компоненты-обертки (RoleGuard.tsx - для проверки прав доступа)
    │   ├── layouts/        - Layout'ы страниц (UserLayout, AdminLayout, AuthorLayout)
    │   ├── CartSidebar.tsx - Боковая панель корзины
    │   ├── Header.tsx      - Шапка сайта

    ├── context/            - Глобальное состояние приложения (React Context API)

    │   └── CartContext.tsx - Управляет логикой корзины покупок (загрузка, добавление, удаление, оплата товаров)
    ├── pages/              - Страницы приложения (разделены по ролям)
    │   ├── admin/          - Панель администратора (Dashboard, Moderation, Support)

    ├── App.tsx             - Корневой компонент

    └── routes.tsx          - Настройка маршрутизации (React Router)

2. СЕРВЕРНАЯ ЧАСТЬ (Backend)
---------------------------------------------------------
Путь: d:\_учеба\веб-приложения\nidshop\server-dotnet
Стек: ASP.NET Core Web API, Entity Framework Core (EF Core), PostgreSQL.
Архитектура: Многослойная (API -> Application -> Domain -> Infrastructure)

Полная структура папок и ключевых файлов:
server-dotnet/
├── API/
│   └── Controllers/        - Обработчики входящих HTTP запросов
│       ├── AdminController.cs  - Логика администрирования
│       ├── AuthController.cs   - Авторизация, регистрация, обновление профиля, работа с тикетами
│       ├── AuthorController.cs - Взаимодействие авторов (статистика, публикация дизайнов)
│       ├── CartController.cs   - Доступ к корзине пользователя через API
│       └── MarketController.cs - Получение списка активных товаров для витрины
├── Application/
│   ├── DTOs/               - Data Transfer Objects (объекты для передачи данных между слоями)
│   ├── Interfaces/         - Интерфейсы сервисов бизнес-логики
│   └── Services/           - Реализация бизнес-логики (здесь находятся алгоритмы)
│       ├── AuthService.cs      - Хеширование паролей, генерация JWT токенов
│       ├── AuthorService.cs    - Конвертация пользователя в автора, подсчет статистики
│       ├── CartService.cs      - Управление корзиной как заказом со статусом "New"
│       └── MarketService.cs    - Сборка данных для витрины магазина
├── Domain/
│   ├── Entities/           - Модели базы данных
│   │   ├── AppUser.cs          - Пользователь
│   │   ├── AuthorStatistic.cs  - Финансовая статистика автора
│   │   ├── BaseProduct.cs      - Базовый товар (футболка, худи - заготовка для принта)
│   │   ├── Design.cs           - Дизайн (принт), хранит JSON с настройками слоев
│   │   ├── MarketItem.cs       - Выставленный на продажу товар (BaseProduct + Design)
│   │   ├── Order.cs            - Заказ (корзина, если статус New)
│   │   ├── OrderItem.cs        - Позиция в заказе
│   │   └── SupportTicket.cs/SupportMessage.cs - Тикеты техподдержки
│   └── Enums/              - Перечисления (UserRole, OrderStatus, ProductType, DesignStatus)
├── Infrastructure/
│   ├── Data/               - Взаимодействие с базой данных
│   │   └── ApplicationDbContext.cs - Конфигурация EF Core, маппинг сущностей в таблицы
│   └── Security/           - Безопасность
│       └── PasswordHasher.cs - Хеширование паролей с использованием соли (PBKDF2-SHA256)
├── Migrations/             - Файлы миграций базы данных Entity Framework
├── appsettings.json        - Конфигурация (строка подключения к БД, секретный ключ JWT)
└── Program.cs              - Точка входа в backend. Настройка сервисов (DI), JWT, ролей, CORS и миграций/сидирования БД.

3. АЛГОРИТМЫ И СХЕМЫ ВЗАИМОДЕЙСТВИЯ (КЛЮЧЕВЫЕ БЛОКИ)
---------------------------------------------------------

1. Алгоритм Авторизации и Контроля Доступа (RBAC)
-------------------------------------------------
Сценарий: Пользователь логинится на сайте.
- [Client] Отправляет email и пароль на сервер.
- [Server / AuthService] Проверяет существование пользователя. Использует класс `PasswordHasher` для сверки хеша пароля.
- [Server] Если все верно, генерирует JWT-токен, в который вшиты (claims) ID пользователя и его текущая Роль (`Role`).
- [Client] Сохраняет токен и User Data в `localStorage`. В компоненте `routes.tsx` реализованы проверки через `<RoleGuard>`. Если пользователь с ролью "User" пытается зайти на `/author/*` или `/admin/*`, роутер его не пускает или показывает 404/ошибку.

2. Алгоритм Корзины (Cart System)
-------------------------------------------------
Сценарий: Пользователь нажимает "Добавить в корзину".
Особенность: Корзина на бэкенде реализована не как отдельная таблица/кэш, а как незавершенный ЗАКАЗ.
- [Server / CartService] При запросе добавления в корзину:
  1. Ищет у пользователя заказ со статусом `OrderStatus.New`.
  2. Если такого заказа нет — создает новый пустой заказ (со статусом `New`).
  3. Ищет `MarketItem` (готовый к продаже товар).
  4. Проверяет, есть ли такой товар нужного размера в этом заказе. Если есть — увеличивает `Quantity` (количество). Если нет — добавляет новый `OrderItem`.
  5. Пересчитывает `TotalAmount` для заказа.
Сценарий: Оплата (Checkout).
- Модальное окно или кнопка вызывает Checkout, статус корзины (`Order`) меняется с `New` на `Printing`.
- Попутно `CartService` обновляет статистику продаж авторам тех дизайнов, вещи с которыми были куплены, и увеличивает баланс (`AuthorStatistic`).

3. Алгоритм создания кастомного мерча и работы Авторов
-------------------------------------------------
Сценарий: Пользователь становится автором и выкладывает свой Дизайн на футболке.
- [Server / AuthorService] У пользователя меняется `Role` на `Author` и задается уникальный `AuthorNickname`. В БД создается привязанная сущность `AuthorStatistic` для учета его заработка.
- [Client / CreateMerchPage] Пользователь в редакторе (Fabric.js или кастомные слои) накладывает изображения на 3D/2D макет базового продукта (например, T-Shirt Black).
- [Server / AuthorController - Publish] Формируется сущность `Design`, где конфигурация принта сохраняется в `LayersData` (тип `jsonb` в PostgreSQL).
- После публикации "Дизайна", для него создается `MarketItem`, который связывает конкретный `Design` и `BaseProduct` (базовую черную футболку), и ему устанавливается `FinalPrice`. После этого данный товар начинает отдаваться в Market API.

4. Хранение данных
-------------------------------------------------
- PostgreSQL используется в качестве основной СУБД.
- В EF Core (через `ApplicationDbContext`) настроены связи EntityFramework, например "Cascade Delete". Если удалить Автора, удалятся все его Дизайны.
- Списки слоев дизайна (для восстановления редактора) хранятся в эффективном бинарном формате `JSONB` напрямую в колонке таблицы `designs`.
- Используются "Native Enums" в PostgreSQL, с трансляцией из C#-enums (UserRole, и т.д.).
